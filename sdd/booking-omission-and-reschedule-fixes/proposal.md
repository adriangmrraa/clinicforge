# Proposal: Booking Omission and Reschedule Fallback Fixes

## Intent

Resolve the two critical bugs in the booking and rescheduling workflows of the AI Agent:
1. Ensure the agent executes `book_appointment` immediately upon DNI submission for a pre-selected/reserved slot, without getting distracted by clinical descriptions or incorrectly triggering migration rules.
2. Ensure the agent gracefully falls back to offering a new booking when a reschedule request is made but the patient has no upcoming appointments registered.

---

## Scope

### In Scope
1. Update `build_system_prompt()` in `orchestrator_service/main.py`:
   - Add a strict `REGLA DE CONFIRMACIÓN CON DNI (CRÍTICA)` in the booking/admission sections.
   - Add a fallback for empty/past turn lists under the `REPROGRAMAR TURNO` section.
2. Update `BookingAgent` system prompt in `orchestrator_service/agents/specialists.py`:
   - Add `REGLA DE CONFIRMACIÓN CON DNI (CRÍTICA)` to enforce immediate tool execution.
   - Add fallback instructions for rescheduling when `list_my_appointments` returns no upcoming appointments.

### Out of Scope
- Modifying database schemas or models.
- Modifying the underlying APIs for booking, canceling, or rescheduling (`book_appointment`, `cancel_appointment`, `reschedule_appointment`).
- Implementing new tools or changing agent graph nodes.

---

## Approach

### 1. Booking Omission Fix: Enforce DNI Confirmation Rule
We will add `REGLA DE CONFIRMACIÓN CON DNI (CRÍTICA)` in both the solo agent prompt (generated by `main.py`) and the multi-agent `BookingAgent` prompt (`specialists.py`).

#### Rule Definition:
```text
⚠️ REGLA DE CONFIRMACIÓN CON DNI (CRÍTICA E INQUEBRANTABLE):
Cuando el paciente ya eligió/seleccionó un slot de turno (el slot ya está pre-reservado con confirm_slot o en proceso de reserva) y proporciona su DNI (o los datos faltantes como nombre y DNI) para confirmar:
1. DEBÉS llamar a `book_appointment` (o `confirm_slot` si aún no se llamó) INMEDIATAMENTE en esa misma interacción, ANTES de escribir cualquier respuesta o saludo.
2. PROHIBIDO responder de forma puramente conversacional o distraerte con datos clínicos (como descripciones de dientes, radiografías, dolor, etc.) o contexto del perfil del paciente antes de realizar la llamada a la tool.
3. PROHIBIDO activar la regla de 'DETECCIÓN DE PACIENTE EXISTENTE SIN DATOS EN SISTEMA (MIGRACIÓN)' en este paso. Si el paciente proporciona su DNI en el contexto de confirmar su turno, NO debés asumir que es un paciente no migrado y NO debés derivar a humano. El flujo de booking debe completarse primero.
```

### 2. Rescheduling Fallback Fix: Gracefully Handle Empty Turn List
We will update the rescheduling flow instructions in both prompts.

#### Prompt Changes in `main.py` (`REPROGRAMAR TURNO (flujo obligatorio)`):
```text
• REPROGRAMAR TURNO (flujo obligatorio):
  R1. Llamá `list_my_appointments` para identificar el turno a reprogramar.
      ⚠️ FALLBACK CRÍTICO: Si `list_my_appointments` devuelve que el paciente NO tiene turnos futuros activos (la respuesta no contiene "PRÓXIMOS:" o está vacía), NO inventes un turno, NO llames reschedule_appointment y NO agendes uno de forma unilateral. Decile al paciente de forma clara: "No encuentro ningún turno agendado a tu nombre en el sistema." Y preguntale si prefiere coordinar un nuevo turno desde cero (si dice que sí, iniciá el flujo de booking normal con check_availability).
  R2. Si tiene turnos, llamá `check_availability` para buscar nuevas opciones (siempre mostrá las 2 opciones al paciente).
  R3. Cuando el paciente elija, llamá reschedule_appointment(original_date="fecha_turno_original", new_date_time="fecha_y_hora_elegida", interpreted_date="YYYY-MM-DD HH:MM").
  R4. Confirmá al paciente: nuevo día, hora y sede. NO llames book_appointment después de un reschedule exitoso.
```

#### Prompt Changes in `specialists.py` (`BookingAgent` prompt):
```text
# CANCELACIÓN Y REPROGRAMACIÓN
- Llamá a `list_my_appointments` primero para identificar el/los turnos futuros del paciente.
- ⚠️ FALLBACK SI NO TIENE TURNOS: Si `list_my_appointments` devuelve que no tiene turnos futuros (lista vacía), NO inventes un turno ni intentes reprogramar o agendar un slot directamente de forma unilateral. Informale al paciente amablemente que no posee turnos agendados en el sistema y ofrecelo coordinar un nuevo turno desde cero (si acepta, iniciá el flujo normal de check_availability).
- Confirmá el turno exacto (día + hora + tratamiento) antes de cancelar/reprogramar.
- Reprogramación = usar `reschedule_appointment` (o cancel_appointment + flujo de booking normal si aplica).
```

## Risks

| Risk | Likelihood | Mitigation |
|------|------------|------------|
| Prompt size exceeds LLM attention limit. | Low | Both prompts are already large, but the added rules are compact and targeted to state transitions. |
| Agent still triggers migration if DNI matches a patient with prior history. | Low | The rule explicitly bans migration trigger when the user is in a booking confirmation turn. |

## Success Criteria
- [ ] A patient in a pre-reserved/selected slot provides their DNI along with a clinical description (e.g. "mi DNI es 12345678, me duele la muela derecha") and the agent calls `book_appointment` instead of responding textually or triggering the migration handoff.
- [ ] A patient requests to reschedule but has no active future appointments. The agent calls `list_my_appointments`, realizes the list is empty/has no upcoming appointments, informs the patient, and asks if they want to book a new appointment instead of hallucinating/booking unilaterally.
