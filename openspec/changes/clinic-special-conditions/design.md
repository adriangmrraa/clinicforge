## Architecture Decisions

### D1: 7 individual columns vs single JSONB blob

**Decision**: Mix — boolean/integer primitives as individual columns, open-ended protocol map as JSONB.

`accepts_pregnant_patients`, `accepts_pediatric`, `requires_anamnesis_before_booking` are BOOLEANs because:
- They are directly queryable without JSONB path expressions
- They have clear semantic meaning as first-class schema elements
- They map cleanly to individual toggle UI controls

`high_risk_protocols` is JSONB because:
- The condition keys are open-ended (clinic can add custom conditions)
- Each condition has a nested multi-field structure
- The set of conditions varies across clinics

`pregnancy_restricted_treatments` is JSONB array (not a FK relation table) because:
- Treatment codes are already strings used throughout the system
- The list is short (rarely > 5 items)
- A separate junction table would be disproportionate for this feature

---

### D2: `_format_special_conditions()` — pure function, not method

**Decision**: Standalone pure function in `main.py`, near the other `_format_*` formatters (`_format_insurance_providers`, `_sanitize_ad_context`).

**Signature**:
```python
def _format_special_conditions(
    tenant_data: dict,
    treatment_name_map: dict[str, str] | None = None,
) -> str:
```

Where `treatment_name_map` is a `{code: display_name}` dict built from the tenant's `treatment_types` query (already available in `buffer_task.py`). If `None`, treatment codes are shown as-is.

**Returns**: Empty string `""` if nothing is configured. Never `None`.

**Why pure function**: testable without DB, no side effects, consistent with existing formatter pattern. The function is called inside `build_system_prompt()` with the pre-fetched tenant data passed as a plain dict.

---

### D3: Integration with `triage_urgency` — context injection approach

**Decision**: Inject special condition policy into the system prompt (not into the tool logic).

The `triage_urgency` tool already runs within the full system prompt context. When `_format_special_conditions()` generates a non-empty block, the agent has the policy text during the entire conversation — including when it calls `triage_urgency`.

The triage tool itself is NOT modified. Instead, the `CONDICIONES ESPECIALES` block in the prompt includes the instruction:

```
INSTRUCCIÓN POST-TRIAGE: Si triage_urgency identifica que el paciente tiene una condición de alto riesgo
(basado en síntomas que el paciente describe), verificar si esa condición tiene protocolo configurado
y reestablecer la política de la clínica al comunicar el resultado del triage.
```

**Why not modify `triage_urgency` directly**: the tool uses `current_tenant_id` context var and could theoretically do a DB lookup. However, this would couple triage logic to business policy lookup, increasing tool complexity and latency. The prompt instruction approach is simpler and sufficient for this use case.

---

### D4: Anamnesis gate — prompt instruction only, no `book_appointment` code change

**Decision**: The anamnesis gate is a prompt-level instruction, not a code-level guard in `book_appointment`.

The `book_appointment` tool already has an `anamnesis_url` parameter (passed when the patient has a pending anamnesis). The prompt instruction tells the agent to check `patient_context` for `anamnesis_completed_at` before calling `book_appointment`.

The prompt gate instruction (generated by `_format_special_conditions` when `requires_anamnesis_before_booking = true`):
```
ANAMNESIS GATE (ACTIVO):
Si el paciente menciona una condición de alto riesgo Y su contexto indica que no tiene anamnesis completada
→ Enviar el link de anamnesis ({anamnesis_url}) ANTES de llamar a book_appointment.
→ Excepción: si triage_urgency retorna "emergency" o "high" → omitir el gate, ofrecer turno inmediato.
```

**Why not code-level**: modifying `book_appointment` would break its clean API and require passing tenant policy to the tool. The prompt-driven approach is backward-compatible and keeps tools focused on their single responsibility.

---

### D5: Migration number — `036`

The current Alembic head is `035` (`035_add_payment_financing_config.py`). Migrations `033` (bot_name), `034` (insurance), `035` (payment_financing) are prior sibling changes. This change's migration is `036`.

The brief referenced the folder name convention (`clinic-special-conditions`), not the migration sequence number. The actual Alembic `revision = "036"`, `down_revision = "035"`.

---

### D6: Frontend — textarea for `high_risk_protocols` (v1), visual editor deferred

**Decision**: v1 uses a structured form for pregnancy and pediatric (simple fields), but `high_risk_protocols` uses a dynamic list of condition cards — each card rendered from a JavaScript object, with an "Add condition" button.

**Rationale**: A raw JSON textarea would expose JSONB syntax errors to non-technical clinic staff. A card-per-condition approach (with labeled fields) is usable without technical knowledge. This is still simpler than a full drag-and-drop editor.

**Card structure per condition**:
- Text input: condition name (e.g., "diabetes")
- Checkbox: Requiere clearance médico
- Checkbox: Requiere llamada previa
- Textarea: Notas para el paciente
- Tag input (comma-separated): Tratamientos restringidos (treatment codes)
- Trash icon: remove condition

#### 4A: New `_format_special_conditions()` function

Add near the other `_format_*` helpers (before `build_system_prompt`):

```python
def _format_special_conditions(
    tenant_data: dict,
    treatment_name_map: dict | None = None,
) -> str:
    """
    Genera el bloque de 'Condiciones Especiales' para el system prompt.
    Retorna "" si ninguna condición está configurada (para no inflar el prompt).
    NUNCA genera consejo médico — solo reestablece la política de la clínica.
    """
    accepts_pregnant = tenant_data.get("accepts_pregnant_patients")
    pregnancy_restricted = tenant_data.get("pregnancy_restricted_treatments") or []
    pregnancy_notes = (tenant_data.get("pregnancy_notes") or "").strip()
    accepts_pediatric = tenant_data.get("accepts_pediatric")
    min_age = tenant_data.get("min_pediatric_age_years")
    pediatric_notes = (tenant_data.get("pediatric_notes") or "").strip()
    high_risk = tenant_data.get("high_risk_protocols") or {}
    requires_anamnesis_gate = tenant_data.get("requires_anamnesis_before_booking", False)

    # If truly nothing configured: return empty to avoid token waste
    pregnancy_configured = (
        accepts_pregnant is False
        or pregnancy_restricted
        or pregnancy_notes
    )
    pediatric_configured = (
        accepts_pediatric is False
        or min_age is not None
        or pediatric_notes
    )
    high_risk_configured = bool(high_risk)

    if not pregnancy_configured and not pediatric_configured and not high_risk_configured and not requires_anamnesis_gate:
        return ""

    lines = [
        "## CONDICIONES ESPECIALES Y RESTRICCIONES (POLÍTICA DE LA CLÍNICA)",
        "",
        "REGLA DE ORO: NUNCA dar consejo médico. NUNCA decir que algo 'es peligroso' o 'está contraindicado médicamente'.",
        "Solo reestablecé la política de la clínica usando las notas configuradas.",
        "Ante cualquier condición especial: (1) tranquilizar, (2) reestablecer política, (3) ofrecer evaluación, (4) sugerir ficha médica.",
        "",
    ]

    # Pregnancy block
    if pregnancy_configured:
        lines.append("EMBARAZO:")
        if accepts_pregnant is False:
            lines.append("• Según política de la clínica, actualmente no se atienden pacientes embarazadas.")
        else:
            lines.append("• Según política de la clínica, se atienden pacientes embarazadas.")
        if pregnancy_restricted:
            names = []
            for code in pregnancy_restricted:
                name = (treatment_name_map or {}).get(code, code)
                names.append(name)
            lines.append(f"• Tratamientos con restricción durante el embarazo: {', '.join(names)}.")
        if pregnancy_notes:
            lines.append(f"• Mensaje para paciente embarazada: \"{pregnancy_notes}\"")
        lines.append("")

    # Pediatric block
    if pediatric_configured:
        lines.append("PEDIATRÍA:")
        if accepts_pediatric is False:
            lines.append("• Según política de la clínica, no se atienden pacientes pediátricos actualmente.")
        else:
            lines.append("• Según política de la clínica, se atienden pacientes pediátricos.")
            if min_age is not None:
                lines.append(f"• La clínica atiende pacientes desde los {min_age} años.")
        if pediatric_notes:
            lines.append(f"• Nota adicional: \"{pediatric_notes}\"")
        lines.append("")

    # High-risk protocols block
    if high_risk_configured:
        lines.append("CONDICIONES DE ALTO RIESGO (protocolos de la clínica):")
        for condition, protocol in high_risk.items():
            if isinstance(protocol, dict):
                clearance = protocol.get("requires_medical_clearance", False)
                pre_call = protocol.get("requires_pre_appointment_call", False)
                restricted = protocol.get("restricted_treatments", [])
                notes = (protocol.get("notes") or "").strip()
                parts = [f"  • {condition}:"]
                if clearance:
                    parts.append("requiere clearance médico previo")
                if pre_call:
                    parts.append("requiere llamada del equipo antes del turno")
                if restricted:
                    restricted_names = [(treatment_name_map or {}).get(c, c) for c in restricted]
                    parts.append(f"tratamientos restringidos: {', '.join(restricted_names)}")
                if notes:
                    parts.append(f"Nota: \"{notes}\"")
                lines.append(" — ".join(parts) if len(parts) > 1 else parts[0])
        lines.append("")

    # Anamnesis gate
    if requires_anamnesis_gate:
        lines.append("ANAMNESIS GATE (ACTIVO):")
        lines.append("Si el paciente menciona una condición de alto riesgo Y su contexto muestra anamnesis no completada →")
        lines.append("  Enviar link de anamnesis ANTES de llamar a book_appointment.")
        lines.append("  Explicar: 'Antes de coordinar el turno, necesitamos que completes tu ficha médica.'")
        lines.append("  EXCEPCIÓN: si triage_urgency retorna 'emergency' o 'high' → omitir gate, ofrecer turno de inmediato.")
        lines.append("")

    # Fallback rule for unknown conditions
    lines.append("Si el paciente menciona una condición NO listada arriba:")
    lines.append("  Responder: 'Gracias por avisarnos. Es importante que el profesional lo sepa. Te recomiendo mencionarlo en la consulta y completar tu ficha médica.'")
    lines.append("  NUNCA decir que la condición impide el tratamiento sin que sea política explícita de la clínica.")

    return "\n".join(lines)
```

## Risks

| Risk | Severity | Mitigation |
|------|----------|------------|
| Agent gives medical advice from the condition block | HIGH | Formatter includes the REGLA DE ORO block verbatim. Unit tests verify no medical-advice language in formatter output. The anti-medical-advice requirement is tested independently. |
| Clinic configures incorrect policy (e.g., tells pregnant patients a dangerous treatment is safe) | HIGH | This is the clinic's legal responsibility, explicitly disclaimed in the UI. The agent only parrots what is configured. ClinicForge is not liable for incorrect policy configuration. |
| asyncpg JSONB returned as string for new columns | MEDIUM | Apply `json.loads()` defensively when reading `high_risk_protocols` and `pregnancy_restricted_treatments` from DB row (same pattern used for `working_hours`). |
| Frontend adds new state fields but forgets to send them in PUT body | LOW | Integration test covers the round-trip: set in UI → save → reload → verify values persist. |
| Emergency bypass of anamnesis gate not respected | MEDIUM | Unit test: call `_format_special_conditions` with `requires_anamnesis_before_booking=True`, verify gate instruction includes the EXCEPCIÓN clause. Scenario test SC-5 verifies behavior. |
