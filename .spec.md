# Spec: Importación Masiva de Pacientes via CSV/Excel

## Contexto
La doctora necesita migrar su base de pacientes existente a ClinicForge para que el agente de IA tenga contexto histórico desde el día 1. Los pacientes pueden tener datos incompletos — el agente los completará progresivamente durante las interacciones.

---

## Columnas del archivo (CSV / XLSX)

### Nombres aceptados (case-insensitive, con o sin tildes)

| Columna | Aliases aceptados | Requerido | Notas |
|---------|-------------------|-----------|-------|
| **nombre** | nombre, first_name, name, primer_nombre | **SÍ** | Único dato obligatorio |
| apellido | apellido, last_name, surname, segundo_nombre | No | |
| telefono | telefono, phone, phone_number, celular, tel, whatsapp | No | Si falta → placeholder `SIN-TEL-XXX` |
| dni | dni, documento, document, id_number, cedula | No | Si falta → placeholder `SIN-DNI-XXX` |
| email | email, correo, mail, e-mail | No | |
| fecha_nacimiento | fecha_nacimiento, birth_date, nacimiento, birthday, fecha_nac | No | Acepta DD/MM/AAAA o AAAA-MM-DD |
| obra_social | obra_social, insurance, seguro, prepaga, mutual | No | |
| ciudad | ciudad, city, localidad, domicilio | No | |
| notas | notas, notes, observaciones, comentarios | No | Texto libre → campo `notes` del paciente |

### Mínimo y máximo

- **Mínimo**: solo la columna `nombre` — todo lo demás es opcional. Placeholders se generan automáticamente para phone y DNI faltantes.
- **Máximo**: las 9 columnas listadas arriba. Columnas adicionales que no matcheen ningún alias serán ignoradas silenciosamente.
- **Límite de filas**: 1000 por archivo. Si se excede → error inmediato sin procesar.

### Ejemplo de archivo mínimo

```
nombre,telefono
María López,+5491155551234
Juan Pérez,
Carlos Gómez,+5491155559876
```

### Ejemplo de archivo completo

```
nombre,apellido,telefono,dni,email,fecha_nacimiento,obra_social,ciudad,notas
María,López,+5491155551234,35789456,maria@email.com,15/03/1990,OSDE,CABA,Alérgica a penicilina
Juan,Pérez,,,,,,Neuquén,Paciente histórico sin contacto
Carlos,Gómez,+5491155559876,40123456,,,Swiss Medical,,
```

## Arquitectura

### Backend — Flujo en 2 pasos

#### Paso 1: `POST /admin/patients/import/preview`

Sube el archivo y devuelve una previsualización SIN insertar nada.

**Request:** multipart/form-data con el archivo (.csv o .xlsx)

**Response:**
```json
{
  "total_rows": 150,
  "valid_new": 120,
  "duplicates": 25,
  "errors": 5,
  "duplicate_details": [
    {
      "row": 3,
      "csv_name": "María López",
      "csv_phone": "+5491155551234",
      "existing_id": 42,
      "existing_name": "Maria Lopez",
      "existing_phone": "+5491155551234",
      "has_new_data": true
    }
  ],
  "error_details": [
    { "row": 7, "reason": "Falta el campo 'nombre'" }
  ],
  "preview_rows": [
    { "row": 1, "first_name": "María", "last_name": "López", "phone_number": "+5491155551234", "status": "new" },
    { "row": 2, "first_name": "Juan", "last_name": "Pérez", "phone_number": "SIN-TEL-001", "status": "new" }
  ]
}
```

**Lógica:**
1. Validar extensión (.csv / .xlsx) y tamaño (max 1000 filas)
2. Parsear archivo (UTF-8, fallback latin-1)
3. Mapear headers a aliases
4. Por cada fila:
   - Si `first_name` vacío → agregar a `error_details`
   - Si `phone_number` vacío → generar placeholder `SIN-TEL-{NNN}` (NNN = counter secuencial)
   - Si `dni` vacío → generar placeholder `SIN-DNI-{NNN}`
   - Normalizar phone (strip spaces, guiones, paréntesis)
   - Buscar duplicados por phone_number en DB del tenant
   - Clasificar como `new`, `duplicate`, o `error`
5. Devolver preview sin insertar

#### Paso 2: `POST /admin/patients/import/execute`

Ejecuta la importación real con la decisión del usuario sobre duplicados.

**Request:**
```json
{
  "file_hash": "abc123",
  "duplicate_action": "skip" | "update",
  "rows": [ ... parsed rows from preview ... ]
}
```

**Lógica:**
- `duplicate_action = "skip"` → insertar solo los `new`, ignorar duplicados
- `duplicate_action = "update"` → insertar `new` + para duplicados, hacer `UPDATE` con `COALESCE` (solo completar campos vacíos en DB, nunca sobreescribir datos existentes)
- Todos los inserts usan `tenant_id` del JWT (Regla de Oro)
- `status = 'active'` para nuevos pacientes

**Response:**
```json
{
  "imported": 120,
  "updated": 25,
  "skipped": 0,
  "errors": 5
}
```

**Nota sobre file_hash**: En la preview se genera un hash del archivo parseado y se guarda en Redis (TTL 10 min) con las filas ya procesadas. El execute recibe ese hash y lee de Redis para no parsear dos veces. Si Redis no está disponible, el frontend re-envía el archivo.

**Simplificación alternativa**: El frontend guarda los `preview_rows` en state y los envía directamente en el execute, sin necesidad de Redis ni file_hash. Esto es más simple y funciona igual.

### Frontend — Modal de importación

**Archivo:** `frontend_react/src/views/PatientsView.tsx`

#### Flujo visual:

**Estado 1 — Selección de archivo**
- Botón "Importar Pacientes" en el header (junto a "Nuevo Paciente")
- Modal con zona drag & drop + click para seleccionar archivo
- Muestra formatos aceptados (.csv, .xlsx) y límite (1000 filas)
- Tabla con formato de columnas esperado (versión compacta)
- Botón "Previsualizar" → llama al endpoint preview

**Estado 2 — Previsualización**
- Resumen: N nuevos, N duplicados, N errores
- Si hay duplicados: bloque amarillo con lista y 2 botones:
  - "Omitir duplicados" (skip)
  - "Completar datos faltantes" (update — solo llena campos vacíos)
- Si hay errores: bloque rojo con detalle por fila
- Botón "Confirmar Importación" → llama al endpoint execute

**Estado 3 — Resultado**
- Resumen final: importados, actualizados, omitidos, errores
- Botón "Cerrar" → cierra modal y refresca lista de pacientes

### Tenant Isolation

`tenant_id` viene de `Depends(get_resolved_tenant_id)` — nunca del request, nunca del CSV. Todos los pacientes importados pertenecen al tenant del usuario autenticado.

### Encoding

1. Intentar parsear como UTF-8
2. Si falla (UnicodeDecodeError) → reintentar con `latin-1`
3. Si falla ambos → error "Encoding no soportado"

### Placeholders

- Phone: `SIN-TEL-001`, `SIN-TEL-002`, ... (counter global dentro de la importación, verificando que no exista ya en DB)
- DNI: `SIN-DNI-001`, `SIN-DNI-002`, ... (mismo patrón)
- El agente de IA y el staff pueden actualizar estos valores luego desde la ficha del paciente o durante la interacción por WhatsApp

## Dependencias

- Backend: `openpyxl` (para .xlsx) — agregar a `requirements.txt`
- Frontend: ninguna nueva (FormData nativo + estado React)

## Orden de implementación

1. `requirements.txt`: agregar `openpyxl`
2. Backend: endpoint preview (`POST /admin/patients/import/preview`)
3. Backend: endpoint execute (`POST /admin/patients/import/execute`)
4. Frontend: botón + modal con 3 estados (upload → preview → result)
5. i18n: keys `patients.import_*` en es/en/fr

## Verificación

1. CSV con 5 pacientes completos → preview muestra 5 new → execute importa 5
2. CSV con 2 duplicados (mismo teléfono ya en DB) → preview muestra 3 new + 2 duplicates
   - Skip → 3 importados, 2 omitidos
   - Update → 3 importados, 2 actualizados (solo campos vacíos)
3. CSV con fila sin nombre → error reportado para esa fila, no se importa
4. CSV con fila sin teléfono → placeholder `SIN-TEL-001` generado, se importa OK
5. XLSX funciona igual que CSV
6. Headers en español ("nombre", "telefono") se mapean correctamente
7. Archivo con encoding latin-1 (tildes) → detectado y parseado correctamente
8. Archivo con > 1000 filas → error inmediato "Máximo 1000 filas"
9. Multi-tenant: pacientes de tenant A no aparecen ni se tocan desde tenant B

## Clarificaciones (Resueltas)

1. **Pacientes sin teléfono/DNI**: Se generan placeholders automáticos (`SIN-TEL-XXX`, `SIN-DNI-XXX`) para que se carguen igual. El staff o el agente de IA completarán el dato real luego.

2. **Duplicados**: Flujo en 2 pasos. La previsualización muestra duplicados detectados y el usuario elige entre "Omitir" (skip) o "Completar datos faltantes" (update con COALESCE — nunca sobreescribe datos existentes).

3. **Límite de filas**: 1000 filas máximo por archivo. Error inmediato si se excede.

4. **Encoding**: UTF-8 primero, fallback automático a latin-1.

5. **Campo notes**: El contenido de la columna "notas/observaciones" del CSV va al campo `notes` (texto libre) del paciente, no a `medical_history` (JSONB).
